# Spring Ai

## 一：框架说明

SpringAI 是 Spring 官方推出的面向 Java 生态的 AI 应用开发框架，提供了标准化的大模型接入与 AI 应用编排能力。它兼容主流商用大模型与本地开源大模型，内置对话管理、向量嵌入、文档处理、函数调用等通用 AI 组件，可与 Spring Boot 项目原生集成，屏蔽了不同大模型厂商的接口差异，避免了传统方案中手动封装 HTTP 接口、跨语言服务运维的额外成本。

在本项目中，我们基于 SpringAI 的统一 API 完成大模型基座对接、会话生命周期管理、RAG 全链路编排以及业务函数的注册与调度，整体技术栈与现有 Java 后端体系完全对齐，开发与运维效率更高。

大白话：

它就是 Spring 官方给 Java 后端做的 AI 开发工具包，地位相当于 MyBatis 对接数据库。以前 Java 对接大模型要自己写请求、解析返回、处理各种兼容问题，现在它全封装好了，还自带做对话、知识库、调用接口的现成功能，不用单独搭 Python 服务，Java 项目引个依赖就能直接开发 AI 功能。

### 二、多轮对话能力实现

#### 官方面试表述

我们基于 SpringAI 的`ChatMemory`对话记忆组件实现多轮对话能力，以会话 ID 为维度对用户对话历史做隔离存储，每轮用户提问时，框架自动将历史对话上下文与当前问题拼接后提交给大模型，保障语义理解的连贯性，支持指代消解、意图延续等多轮交互场景。

针对长会话的 token 膨胀问题，我们采用滑动窗口 + 对话摘要的混合优化策略：保留最近 5-8 轮的完整对话记录，对超出窗口的历史对话通过大模型生成精简语义摘要并入上下文，在保证上下文语义完整性的同时，有效控制单轮请求的 token 长度，降低大模型调用成本。

结合信贷业务场景，我们在系统提示词中增加了业务指代约束，进一步提升多轮咨询中业务术语与上下文指代的识别准确率。

#### 大白话通俗理解

就是让客服 “不健忘”—— 用户先问 “贷款需要什么条件”，再问 “那我征信有逾期能办吗”，它能知道 “那” 指的是刚才的贷款条件，不会聊两句就断片。

实现逻辑就是给每个用户存一份聊天记录，每次提问都把之前的聊天记录一起发给大模型。但聊天记录不能无限存，不然费钱还超过大模型的长度限制，所以我们只存最近几轮完整内容，更早的就总结成一句话梗概，既不丢关键信息又控制成本。

### 三、本地 RAG 知识库搭建

#### 官方面试表述

我们基于 SpringAI 的文档处理与向量嵌入能力搭建了本地化 RAG（检索增强生成）知识库，核心解决两个问题：一是通用大模型不掌握信贷平台专属业务规则，回答准确率不足；二是大模型存在生成幻觉，金融场景存在合规风险；同时全链路本地化部署满足金融数据不出域的合规要求。

整体实现分为两个核心阶段：

1. **知识库构建阶段**：将信贷产品手册、业务流程规范、合规应答话术、常见问题 FAQ 等静态业务文档，通过 SpringAI 的`TextSplitter`组件按语义粒度分块切片，采用重叠分块策略避免语义断裂；随后通过本地部署的嵌入模型将文本块转换为向量，持久化存储至本地向量数据库（PGVector/Milvus），完成知识库的初始化入库。
2. **检索生成阶段**：用户提问时，首先将用户问题向量化，在向量库中执行语义相似度检索，召回 Top3-Top5 最相关的业务知识片段；再将系统提示词、召回的业务知识、用户问题一同组装为 Prompt 提交给大模型，通过提示词工程严格约束大模型仅基于召回的业务知识生成回答，未知问题统一引导转人工客服，从机制上保障业务回答的准确性与合规性。

#### 大白话通俗理解

RAG 就是 “先查资料再回答”，专门防止大模型瞎编。大模型本身只有通用知识，不懂咱们公司具体的贷款利率、申请流程、合规话术，还容易胡说八道，金融行业说错话风险很高，而且业务数据敏感不能传给第三方。

所以我们把所有信贷业务资料都存在自己的服务器上：先把长文档切成小段，转成 “语义身份证” 存在本地向量库里；用户提问时，先去库里搜最相关的几段资料，再让大模型照着资料回答，不知道就说不知道，既准确又合规，后续业务规则更新了，直接往库里加新文档就行。

### 四、业务接口关联（Function Calling）实现

#### 官方面试表述

我们基于 SpringAI 的 Function Calling（函数调用）能力，实现智能客服与信贷业务系统的接口联动，解决静态知识库无法覆盖实时、个性化数据查询的问题，支持用户账单查询、贷款额度测算、还款日查询等动态业务场景。

具体实现流程如下：

1. 首先将信贷平台的业务接口封装为标准化工具函数，在 SpringAI 中注册函数元数据，包括函数功能描述、入参字段定义、数据类型说明；
2. 用户提问时，大模型自主判断问题类型：静态知识类走 RAG 检索流程，实时数据类则判定需要调用工具函数，并返回指定的函数名称与解析后的入参；
3. SpringAI 框架自动触发对应 Java 方法执行，调用内部业务接口获取实时数据，调用过程中内置权限校验与参数校验，保障接口安全；
4. 最终将接口返回的业务数据与原始问题一同提交给大模型，由大模型整合生成自然语言回答返回给用户，整个调用过程对用户无感知。

#### 大白话通俗理解

知识库只能查固定的规则，比如 “贷款申请需要什么材料”，但用户问 “我还剩多少钱没还”“我最多能借多少” 这种每个人不一样的实时数据，就得调用咱们自己的业务系统接口。

我们把查账单、算额度这些接口做成一个个 “工具” 告诉大模型，大模型遇到需要查数据的问题，就会返回 “我要调用查账单的工具，参数是用户 ID”，SpringAI 自动去调用接口拿到真实数据，再让大模型把数据整理成人话回复用户，全程用户就像跟真人客服对话一样，不用自己去 APP 里查。
