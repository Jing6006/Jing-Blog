# 一、数据一致性（面试正式定义 + 大白话理解）

## 面试标准回答

数据一致性，指**多个数据表、多个服务、多个数据库之间有关联的业务数据，逻辑必须匹配、状态自洽**，要么全部修改成功，要么全部失败回滚，不能出现一部分更新成功、另一部分更新失败的错乱脏数据。

在分布式系统里分为三个级别：

1. **强一致性**：数据修改完成后，所有节点立刻同步最新数据，任何查询都能读到最新结果，实时性最强；
2. **弱一致性**：数据更新后，短暂时间内不同节点读取的数据可能不一致；
3. **最终一致性**：允许短时间数据不一致，经过一段时间同步、补偿之后，所有数据最终达到统一状态，是互联网分布式最常用方案。

## 大白话通俗理解

一件事牵扯好几处数据改动，必须**同生共死**，不能改一半卡住。

举例（你的信贷还款场景）：用户还款成功，要做三件事：扣账户余额、更新账单为已结清、新增还款流水。

- 一致情况：三个操作全部成功 / 全部失败回滚；
- 不一致问题：钱扣走了，但账单没改、流水没生成，账目对不上，就是典型数据不一致。

## 不一致产生原因

1. 单机多表：程序中途异常、代码报错，事务中途中断；
2. 分布式场景：跨服务调用超时、网络中断、下游服务宕机，跨库操作没法天然原子执行。

------

# 二、怎么解决？分「单机场景」和「分布式场景」两套方案（面试必背）

## 1、单机单库场景（最简单）

### 解决方案：数据库本地 ACID 事务

把一组关联的增删改 SQL 放到同一个数据库事务中：

- 所有 SQL 全部执行无误，统一提交事务；

- 任意一条语句报错，整体全部回滚，保证多表数据同步。



  适用：所有操作都在同一个 MySQL 库内部。

## 2、分布式多服务 / 多库场景（重点，面试官最爱深挖）

### 方案① TCC 事务（强一致性方案）

三段式设计，侵入业务代码，适合资金、扣款等对实时一致性要求极高的核心金融业务

1. **Try 预留阶段**：检查资源、冻结占用资金 / 库存，做预校验；

2. **Confirm 确认阶段**：全部服务 Try 都成功，正式提交执行业务，完成数据修改；

3. **Cancel 取消阶段**：任意节点 Try 失败，所有服务执行回滚、释放预留资源。



   优点：实时强一致；缺点：开发量大、代码侵入高、维护复杂。

### 方案② 可靠消息最终一致性（项目最常用，适配你的 MQ 知识）

依靠 MQ 事务消息实现，绝大多数异步业务首选

1. 生产者先执行本地事务，同时发送**半消息**，下游消费者暂时收不到这条消息；
2. 本地事务执行成功，正式提交消息，下游消费执行业务；本地事务失败，则撤回这条消息；
3. 如果服务中途宕机，MQ 会主动回查生产者本地事务状态，判断提交还是取消消息；
4. 消费端处理失败自动重试，多次失败转入死信队列人工排查，最终上下游数据对齐。

### 方案③ 最大努力通知

多用于对接第三方外部系统（支付回调、短信推送、对账通知）

通过多次重试回调 + 定时核对，尽可能把通知推送成功；容忍极小概率失败，配套人工对账兜底，不做强事务约束。

### 方案④ 定时对账补偿（兜底必备，所有项目通用）

不管用上面哪种分布式方案，都会额外定时跑对账任务：

定时比对上下游、多库之间业务流水、金额数据，发现数据差额、状态不一致，自动触发补单、冲正修复，作为数据一致性最后的兜底防线。

# 三、1 分钟精简背诵版（面试直接口述）

数据一致性就是关联的多份数据要同步匹配，不能改一半出错。

单机场景直接用数据库本地事务保证；分布式场景分四种方案：金融核心强一致需求用 TCC 三段式事务；大部分异步业务采用 MQ 可靠消息实现最终一致性；第三方对接场景用最大努力通知；最后统一搭配定时对账程序兜底，修复异常数据，保障整体数据一致。
