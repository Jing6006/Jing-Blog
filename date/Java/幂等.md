# 一、幂等是什么（面试标准回答 + 大白话）

## 官方面试表述

幂等性指**同一个请求，执行 1 次，或是重复执行 N 次，最终业务结果完全一致，不会产生额外副作用**。

出现重复请求常见诱因：前端重复点击按钮、接口调用超时自动重试、MQ 消息重复投递、第三方支付回调重复推送。如果接口没有幂等设计，会出现重复扣款、重复生成订单、重复插入多条流水等脏数据问题。

## 大白话

一件事不管你发起多少次，效果永远只等于做一次。

举个例子：还款扣款接口，网络卡顿你点 3 次提交；有幂等就只扣 1 次钱、生成 1 条还款记录；没幂等就扣 3 次钱，账目直接出错。

# 二、五种解决方案（面试逐条口述，带适用场景，贴合你的信贷项目）

## 方案 1：数据库唯一索引（最简单，新增类场景首选）

### 原理

给业务唯一编号（订单号、还款流水号、消息 ID）建立数据库唯一约束。重复插入相同编号数据时，数据库直接报唯一冲突异常，拦截重复数据入库。

### 适用场景

新增订单、新增还款流水这类插入型业务。

## 方案 2：一次性 Token 机制（专治前端重复提交）

### 原理

1. 用户进入提交页面，先向后端申请一个唯一一次性 Token 并存入 Redis；
2. 用户提交请求时必须携带这个 Token；
3. 后端先校验 Redis 是否存在该 Token：存在则立刻删除 Token，再执行业务；不存在直接判定重复请求，拒绝执行。

### 适用场景：页面表单提交、防止用户狂点提交按钮。

## 方案 3：先查询后执行业务（接口重试、回调接口最常用）

### 原理

执行业务更新、扣款逻辑之前，先根据业务单号查询这条业务当前状态；

如果查询结果显示**已处理完成**，直接返回成功，不再重复执行核心逻辑；只有未处理状态才往下走。

### 适用场景：支付回调接口、第三方推送通知、超时重试接口（你的信贷还款回调就是典型场景）。

## 方案 4：状态机控制（有固定流程的订单 / 还款业务）

### 原理

给业务设定固定状态流转链路，比如还款流程：待还款 → 还款中 → 已还款。

只有当前状态满足前置条件，才允许执行操作；重复请求过来时状态已经走完，直接拦截。

### 适用场景：订单状态流转、贷款还款、审批流程类业务。

## 方案 5：Redis 分布式锁（高并发场景防重复操作）

### 原理

同一个业务单号进来，先抢占 Redis 分布式锁，抢到锁才允许执行业务逻辑，执行完毕释放锁；后续重复请求抢不到锁，直接快速返回，避免并发下重复处理。

### 适用场景：高并发扣款、秒杀、多实例部署的接口防重。

# 三、精简背诵版（面试 1 分钟说完，直接背）

幂等就是同一个请求多次执行，结果和执行一次完全相同，避免重复扣款、重复插数据。

解决方式分五种：新增数据用数据库唯一索引防重复插入；前端频繁提交用一次性 Token；支付回调、接口重试采用先查业务状态再处理；订单还款这类流程业务用状态机判断；高并发场景借助 Redis 分布式锁控制同一业务同一时间只处理一次。
