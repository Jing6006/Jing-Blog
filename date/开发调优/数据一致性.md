## 什么是数据一致性

数据一致性是指在分布式系统中，**多个数据源之间的数据在逻辑上保持匹配和自洽的状态**。当一个业务操作涉及多个数据表、多个服务或多个数据库时，这些数据要么全部更新成功，要么全部失败回滚，不能出现部分成功、部分失败的中间状态。

### 一致性级别

根据数据同步的实时性要求，可以分为三个级别：

1. **强一致性**：数据修改后，所有节点立即可见最新数据，任何查询都返回一致结果
2. **弱一致性**：数据更新后，不保证所有节点立即同步，存在短暂的不一致窗口
3. **最终一致性**：允许短时间内数据不一致，但通过补偿机制最终达到一致状态（最常用）

### 典型场景

以支付系统为例，用户完成一笔支付需要：
- 扣减用户账户余额
- 增加商家账户余额
- 生成支付流水记录
- 更新订单状态为已支付

这四个操作必须保持一致性，不能出现"钱扣了但订单没更新"或"流水记录了但余额没变"的情况。

## 数据不一致的产生原因

### 单机场景
- 程序执行中途抛出异常
- 事务未正确提交或回滚
- 数据库连接异常

### 分布式场景
- 服务间网络通信超时
- 下游服务宕机或不可用
- 分布式事务协调失败
- 消息丢失或重复消费

## 单机场景解决方案

### 本地事务（ACID）

在单个数据库内，使用数据库事务机制保证多个操作的原子性：

```java
@Transactional(rollbackFor = Exception.class)
public void processPayment(PaymentRequest request) {
    // 扣减账户余额
    accountService.deductBalance(request.getUserId(), request.getAmount());
    
    // 更新订单状态
    orderService.updateStatus(request.getOrderId(), OrderStatus.PAID);
    
    // 生成支付流水
    paymentRecordService.createRecord(request);
}
```

**适用场景**：所有操作都在同一个数据库实例中

**优点**：简单可靠，数据库原生支持

**局限性**：无法跨数据库、跨服务

## 分布式场景解决方案

### 方案一：TCC 事务

TCC（Try-Confirm-Cancel）是一种补偿型事务方案，将业务逻辑分为三个阶段：

**1. Try 阶段（尝试预留）**
- 检查业务资源是否充足
- 预留或冻结业务资源
- 执行业务预校验

**2. Confirm 阶段（确认提交）**
- 所有参与方 Try 都成功后执行
- 正式提交业务操作
- 使用预留的资源完成最终业务

**3. Cancel 阶段（取消回滚）**
- 任意参与方 Try 失败时执行
- 释放 Try 阶段预留的资源
- 回滚已执行的操作

**示例：转账场景**

```java
// Try 阶段：冻结转出方金额
public boolean tryTransfer(String fromAccount, BigDecimal amount) {
    return accountService.freezeBalance(fromAccount, amount);
}

// Confirm 阶段：扣减冻结金额，增加对方余额
public void confirmTransfer(String fromAccount, String toAccount, BigDecimal amount) {
    accountService.deductFrozenBalance(fromAccount, amount);
    accountService.addBalance(toAccount, amount);
}

// Cancel 阶段：解冻金额
public void cancelTransfer(String fromAccount, BigDecimal amount) {
    accountService.unfreeze(fromAccount, amount);
}
```

**适用场景**
- 金融核心业务
- 资金交易
- 对一致性要求极高的场景

**优点**：强一致性，实时性高

**缺点**：
- 开发成本高，代码侵入性强
- 需要实现预留和补偿逻辑
- 性能开销较大

### 方案二：消息队列最终一致性

利用消息队列的事务消息特性，实现跨服务的最终一致性：

**实现流程**

1. **发送半消息**：业务操作前发送 prepared 状态的消息到 MQ
2. **执行本地事务**：完成本地数据库操作
3. **提交/回滚消息**：根据本地事务结果，提交或回滚消息
4. **消费消息**：下游服务消费消息，执行对应业务逻辑
5. **事务回查**：MQ 定期回查本地事务状态，确保消息不丢失

**示例代码**

```java
// 上游服务：发送事务消息
@Transactional
public void createOrder(Order order) {
    // 发送半消息
    rocketMQTemplate.sendMessageInTransaction(
        "order-topic",
        MessageBuilder.withPayload(order).build(),
        order
    );
}

// 本地事务执行器
@RocketMQTransactionListener
public class OrderTransactionListener implements RocketMQLocalTransactionListener {
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            Order order = (Order) arg;
            // 执行本地事务
            orderService.save(order);
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }
    
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        // 回查本地事务状态
        String orderId = msg.getHeaders().get("orderId");
        boolean exists = orderService.exists(orderId);
        return exists ? RocketMQLocalTransactionState.COMMIT 
                      : RocketMQLocalTransactionState.ROLLBACK;
    }
}

// 下游服务：消费消息
@RocketMQMessageListener(topic = "order-topic", consumerGroup = "inventory-consumer")
public class InventoryConsumer implements RocketMQListener<Order> {
    @Override
    public void onMessage(Order order) {
        // 扣减库存（幂等处理）
        inventoryService.deduct(order.getProductId(), order.getQuantity());
    }
}
```

**适用场景**
- 异步业务场景
- 跨服务调用
- 对实时性要求不高的业务

**优点**
- 业务解耦
- 异步处理，性能好
- 最终一致性有保障

**缺点**
- 有短暂的数据不一致窗口
- 需要考虑消息重复消费（幂等）

### 方案三：最大努力通知

通过定时重试和主动查询，尽最大努力完成通知，允许极小概率失败。

**实现机制**

1. 主动通知：调用下游接口，失败后记录到重试表
2. 定时重试：按照退避策略（如 1分钟、5分钟、30分钟）多次重试
3. 主动查询：下游服务可主动查询上游状态
4. 人工介入：超过最大重试次数后转人工处理

```java
public class NotificationService {
    
    public void notify(String url, NotificationData data) {
        try {
            // 调用下游接口
            restTemplate.postForObject(url, data, String.class);
        } catch (Exception e) {
            // 失败记录到重试表
            retryRecordService.save(url, data, 0);
        }
    }
    
    @Scheduled(cron = "0 */1 * * * ?")
    public void retryFailedNotifications() {
        List<RetryRecord> records = retryRecordService.listPending();
        
        for (RetryRecord record : records) {
            if (record.getRetryCount() >= MAX_RETRY) {
                // 转人工处理
                record.setStatus(RetryStatus.MANUAL);
                continue;
            }
            
            try {
                restTemplate.postForObject(record.getUrl(), record.getData(), String.class);
                record.setStatus(RetryStatus.SUCCESS);
            } catch (Exception e) {
                record.setRetryCount(record.getRetryCount() + 1);
            }
            
            retryRecordService.update(record);
        }
    }
}
```

**适用场景**
- 对接第三方系统
- 支付回调通知
- 短信、邮件发送

**优点**：实现简单，容错性好

**缺点**：无法保证 100% 成功，需要人工兜底

### 方案四：定时对账与补偿

无论采用哪种方案，都应该配合定时对账作为最后的兜底手段。

**对账流程**

1. **数据采集**：定期从各个系统导出业务数据
2. **数据比对**：按照业务规则比对数据是否一致
3. **差异识别**：找出不一致的数据记录
4. **自动补偿**：对于可自动修复的差异，触发补偿流程
5. **人工处理**：复杂场景转人工核查处理

```java
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨2点
public void reconciliation() {
    LocalDate yesterday = LocalDate.now().minusDays(1);
    
    // 1. 获取订单系统数据
    List<Order> orders = orderService.listByDate(yesterday);
    
    // 2. 获取支付系统数据
    List<Payment> payments = paymentService.listByDate(yesterday);
    
    // 3. 比对数据
    for (Order order : orders) {
        Payment payment = payments.stream()
            .filter(p -> p.getOrderId().equals(order.getId()))
            .findFirst()
            .orElse(null);
        
        if (payment == null && order.getStatus() == OrderStatus.PAID) {
            // 订单已支付但没有支付记录，记录差异
            reconciliationService.recordDifference(order, "缺少支付记录");
        } else if (payment != null && !payment.getAmount().equals(order.getAmount())) {
            // 金额不一致
            reconciliationService.recordDifference(order, "金额不一致");
        }
    }
}
```

**适用场景**：所有分布式系统

**优点**：兜底保障，发现并修复遗漏问题

**建议**：
- 对账频率根据业务重要性调整（小时级/天级）
- 保留对账日志，方便追溯
- 重要业务设置告警机制

## 方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| 本地事务 | 强一致 | 高 | 低 | 单库操作 |
| TCC | 强一致 | 低 | 高 | 金融核心业务 |
| 消息队列 | 最终一致 | 高 | 中 | 异步业务 |
| 最大努力通知 | 最终一致 | 中 | 低 | 第三方对接 |
| 定时对账 | 最终一致 | 高 | 中 | 所有场景兜底 |

## 实践建议

1. **优先使用本地事务**：能用单库事务解决的，不要过度设计
2. **根据业务选择方案**：
   - 核心资金业务 → TCC
   - 普通异步业务 → 消息队列
   - 第三方对接 → 最大努力通知
3. **必须有兜底机制**：任何方案都要配合定时对账
4. **考虑幂等性**：所有补偿和重试操作都要保证幂等
5. **完善监控告警**：及时发现数据不一致问题

## 总结

分布式系统的数据一致性是一个复杂的工程问题，没有银弹方案。在实际项目中，需要根据业务特点、一致性要求、性能指标等因素综合考虑，选择合适的方案或组合方案，并通过多层保障机制确保数据的最终一致性。
