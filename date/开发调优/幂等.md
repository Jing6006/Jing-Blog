## 什么是幂等性

幂等性是指**同一个请求执行一次或重复执行多次，最终产生的业务结果完全一致，不会产生额外的副作用**。这是分布式系统和接口设计中的一个重要概念。

### 为什么需要幂等性

在实际业务中，重复请求的场景非常常见：

- 用户网络不稳定，前端重复点击提交按钮
- 接口调用超时后触发自动重试机制
- 消息队列的消息可能被重复投递
- 第三方支付回调可能重复推送

如果接口没有幂等性保障，可能导致：
- 订单重复创建
- 资金重复扣款
- 业务流水重复记录
- 数据不一致

## 五种幂等性实现方案

### 方案一：数据库唯一索引

**实现原理**

在数据库表中对业务唯一标识字段（如订单号、流水号、消息ID）添加唯一索引约束。当重复插入相同标识的记录时，数据库会抛出唯一性约束异常，从而自然地防止重复数据。

```sql
CREATE TABLE `payment_record` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `order_no` VARCHAR(64) NOT NULL UNIQUE KEY,  -- 唯一索引
  `amount` DECIMAL(10,2),
  `status` TINYINT
) ENGINE=InnoDB;
```

**适用场景**
- 订单创建
- 支付流水记录
- 消息消费记录

**优点**：实现简单，数据库层面保证
**缺点**：仅适用于插入操作，不适合更新场景

---

### 方案二：Token 令牌机制

**实现原理**

1. 用户进入提交页面时，后端生成唯一的 Token 并存入 Redis，同时返回给前端
2. 用户提交表单时，携带该 Token
3. 后端接收请求后，先检查 Token 是否存在：
   - 存在：删除 Token，执行业务逻辑
   - 不存在：判定为重复请求，直接返回

```java
// 生成Token
String token = UUID.randomUUID().toString();
redisTemplate.opsForValue().set("submit:token:" + token, "1", 5, TimeUnit.MINUTES);

// 校验Token
Boolean success = redisTemplate.delete("submit:token:" + token);
if (!success) {
    throw new BusinessException("请勿重复提交");
}
// 执行业务逻辑...
```

**适用场景**
- 表单提交
- 订单创建页面
- 防止用户频繁点击

**优点**：彻底防止前端重复提交
**缺点**：需要额外的 Token 生成和校验逻辑

### 方案三：状态查询 + 条件更新

**实现原理**

在执行核心业务逻辑前，先查询业务当前状态：
- 如果已处理完成，直接返回成功结果
- 如果未处理，继续执行业务逻辑

```java
public void processPayment(String orderNo) {
    // 先查询订单状态
    Order order = orderMapper.selectByOrderNo(orderNo);
    if (order.getStatus() == OrderStatus.PAID) {
        // 已支付，直接返回
        return;
    }
    
    // 未支付，执行支付逻辑
    doPayment(order);
}
```

或者使用数据库条件更新：

```sql
UPDATE `order` 
SET `status` = 'PAID', `paid_time` = NOW()
WHERE `order_no` = #{orderNo} 
  AND `status` = 'PENDING'  -- 只有待支付状态才能更新
```

**适用场景**
- 支付回调接口
- 第三方通知接口
- 接口超时重试

**优点**：简单直观，易于理解
**缺点**：存在查询和更新的时间窗口

### 方案四：状态机控制

**实现原理**

为业务设计明确的状态流转规则，只允许从特定状态转换到下一个状态。重复请求到达时，由于状态已变更，自然被拦截。

```java
public class OrderStateMachine {
    public void pay(Order order) {
        if (order.getStatus() != OrderStatus.PENDING) {
            throw new BusinessException("订单状态不允许支付");
        }
        
        // 执行支付并更新状态
        order.setStatus(OrderStatus.PAID);
        orderMapper.updateById(order);
    }
}
```

状态流转链路示例：
```
待支付 → 支付中 → 已支付 → 已完成
         ↓ 超时
       已取消
```

**适用场景**
- 订单流程
- 工单审批
- 贷款申请流程

**优点**：业务清晰，流程可控
**缺点**：需要设计完善的状态机

### 方案五：分布式锁

**实现原理**

使用 Redis 分布式锁，同一业务标识同一时刻只允许一个请求获取锁并执行业务逻辑。其他重复请求获取锁失败，快速返回。

```java
public void processOrder(String orderNo) {
    String lockKey = "order:lock:" + orderNo;
    
    // 尝试获取锁，超时时间30秒
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 30, TimeUnit.SECONDS);
    
    if (!locked) {
        throw new BusinessException("订单处理中，请勿重复提交");
    }
    
    try {
        // 执行业务逻辑
        doBusinessLogic(orderNo);
    } finally {
        // 释放锁
        redisTemplate.delete(lockKey);
    }
}
```

**适用场景**
- 高并发场景
- 多实例部署的服务
- 秒杀、库存扣减

**优点**：强一致性，并发安全
**缺点**：性能开销较大，需处理锁超时、死锁问题

## 方案对比与选择

| 方案 | 适用操作 | 性能 | 复杂度 | 推荐场景 |
|------|---------|------|--------|---------|
| 唯一索引 | 插入 | 高 | 低 | 订单创建、流水记录 |
| Token 机制 | 全部 | 中 | 中 | 表单提交 |
| 状态查询 | 更新 | 中 | 低 | 回调接口、重试场景 |
| 状态机 | 流程类 | 高 | 中 | 订单、审批流程 |
| 分布式锁 | 全部 | 低 | 高 | 高并发、强一致性 |

## 实践建议

1. **优先使用数据库层面的保障**：唯一索引、条件更新等，简单可靠
2. **结合业务特点选择方案**：没有银弹，根据场景选择合适方案
3. **多层防护**：可以组合多种方案，如唯一索引 + 分布式锁
4. **完善的异常处理**：幂等冲突时给出友好的错误提示
5. **监控和告警**：记录重复请求日志，及时发现系统异常

## 总结

幂等性是分布式系统和高可用架构中的基础能力。在设计接口时，需要根据业务特点选择合适的幂等性保障方案，确保系统在各种异常情况下都能保持数据的一致性和正确性。
